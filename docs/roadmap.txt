## Phase 6 — Azure Functions Crons
**Duration**: 2–3 days | **Depends on**: Phase 4 (Function App deployed), Phase 3 (token_budgets + agent_quotas tables)

Picking up from Phase 6.1:

---

### 6.1 — Daily Token Budget Reset (Day 1)
- [ ] `token_quota_reset` — Timer Trigger: `0 0 * * *` (midnight UTC)
- [ ] Logic: creates new `token_budgets` rows for `CURRENT_DATE` for all active users — does NOT delete old rows (they're historical data for the Usage page)
- [ ] Uses `ensure_budget_row()` RPC — inserts with `budget_limit` pulled from `admin_settings.global_daily_token_budget`, or the per-user override if one exists
- [ ] Edge case: if a user has a custom `budget_limit` override set by admin on the Budgets page, preserve that override — don't reset it to the global default
- [ ] Logs to Application Insights: `{event: "token_reset", users_reset: N, timestamp: ...}`
- [ ] Verification: after midnight UTC, query `token_budgets WHERE period = CURRENT_DATE` — every active user should have a fresh row with `tokens_used = 0`

### 6.2 — Monthly Agent Quota Reset (Day 1)
- [ ] `agent_quota_reset` — Timer Trigger: `0 0 1 * *` (1st of every month, midnight UTC)
- [ ] Logic: inserts new `agent_quotas` row for each active user with `period = 'YYYY-MM'` format, all counters at 0
- [ ] Old period rows are kept for historical reporting
- [ ] Logs: `{event: "agent_quota_reset", period: "2026-07", users_reset: N}`
- [ ] Verification: on the 1st, confirm `agent_quotas WHERE period = '2026-07'` exists for all active users with zeroed counters

### 6.3 — Hourly Usage Aggregation (Day 1–2)
- [ ] `usage_aggregation` — Timer Trigger: `0 * * * *` (top of every hour)
- [ ] Purpose: materializes pre-computed stats so the Admin Dashboard Usage page doesn't run expensive aggregate queries against `messages` and `jobs` on every page load
- [ ] Creates/updates a `usage_stats` table (or writes to a `usage_snapshots` table):
  ```
  usage_stats:
    period_hour    TIMESTAMPTZ   — truncated to hour
    user_id        UUID
    model          TEXT          — deployment name (gpt-5.4, gpt-5.4-mini, gpt-5.4-nano, o4-mini)
    tokens_in      BIGINT        — SUM(tokens_input) for that hour
    tokens_out     BIGINT        — SUM(tokens_output) for that hour
    request_count  INT           — COUNT of messages
    agent_type     TEXT          — NULL for chat, 'ocr'/'vision'/etc for agents
    agent_calls    INT           — COUNT of completed jobs by type
  ```
- [ ] Query: aggregates from `messages` and `jobs` WHERE `created_at >= last_run AND created_at < now()`
- [ ] Idempotent: uses `ON CONFLICT (period_hour, user_id, model, agent_type) DO UPDATE` — safe to re-run
- [ ] RLS: only `is_admin()` can SELECT from `usage_stats`
- [ ] Verification: after 1 hour of usage, query `usage_stats` — should have rows matching the chat and agent activity

### 6.4 — Conversation Archiver (Day 2)
- [ ] `conversation_archiver` — Timer Trigger: `0 2 * * *` (2am daily UTC)
- [ ] Logic: `UPDATE conversations SET is_archived = TRUE WHERE updated_at < now() - INTERVAL '90 days' AND is_archived = FALSE`
- [ ] Archived conversations remain in the database and are still queryable — they are simply deprioritized in the sidebar (frontend filters `is_archived = FALSE` by default, with a "Show archived" toggle)
- [ ] Does NOT delete any data — this is a soft archive only
- [ ] Logs: `{event: "conversations_archived", count: N, oldest_archived: "2026-03-27"}`
- [ ] Verification: create a test conversation with `updated_at` manually set to 91 days ago → run archiver → confirm `is_archived = TRUE`

### 6.5 — Model Expiry Monitor (Day 2)
- [ ] `model_expiry_monitor` — Timer Trigger: `0 9 * * *` (9am daily UTC)
- [ ] Reads from Azure App Configuration:
  - `THINK_MODEL_DEPLOYMENT` + `THINK_MODEL_EXPIRY_DATE`
  - `SOLVE_MODEL_DEPLOYMENT` + `SOLVE_MODEL_EXPIRY_DATE`
  - `NANO_MODEL_DEPLOYMENT` + `NANO_MODEL_EXPIRY_DATE`
  - `COMPACTION_MODEL_DEPLOYMENT` + `COMPACTION_MODEL_EXPIRY_DATE`
- [ ] For each deployment:
  - **≤ 30 days to expiry** → log WARNING to Application Insights + send alert (email or webhook to admin)
  - **≤ 7 days to expiry** → log CRITICAL + send urgent alert
  - **Expired (past date)** → auto-swap to fallback deployment name stored in `{MODE}_FALLBACK_DEPLOYMENT` App Config key → log `{event: "model_auto_swap", from: "gpt-5.4", to: "gpt-5.4-fallback"}`
- [ ] After any auto-swap: update the primary deployment key in Azure App Configuration so FastAPI reads the new value on next request — zero downtime, zero redeploy
- [ ] Verification: set a test `THINK_MODEL_EXPIRY_DATE` to yesterday → run monitor → confirm auto-swap occurred and App Config updated

### 6.6 — Conversation Summarizer / Chat Compaction (Day 2–3)
- [ ] `conversation_summarizer` — Timer Trigger: `0 3 * * *` (3am daily UTC)
- [ ] Query: `SELECT id FROM conversations WHERE message_count > {COMPACTION_THRESHOLD} AND (last_compacted_at IS NULL OR last_compacted_at < now() - INTERVAL '7 days')`
- [ ] For each qualifying conversation:
  1. Fetch all messages WHERE `is_archived_msg = FALSE` ordered by `created_at ASC`
  2. If count < `COMPACTION_THRESHOLD` (default 50) → skip (message_count may be stale)
  3. Take oldest 60% of messages → format as structured text
  4. Call GPT-o4-mini (`COMPACTION_MODEL_DEPLOYMENT`) via Azure OpenAI Responses API:
     - System prompt: "Summarize this conversation history concisely. Preserve: all decisions made, key facts, user preferences, ongoing tasks, and any code or data shared. Output as structured paragraphs. Be thorough but compact."
  5. Insert a new `[SUMMARY]` message: `role = 'system'`, `is_summary = TRUE`, `is_archived_msg = FALSE`, `routing_mode = 'summary'`
  6. Mark the old messages: `UPDATE messages SET is_archived_msg = TRUE WHERE id IN (...oldest 60%...)`
  7. Update conversation: `last_compacted_at = now()`
- [ ] Safeguards:
  - Max 20 conversations processed per run (prevent timeout on Function consumption plan — 10 minute max execution)
  - If GPT-o4-mini call fails, skip that conversation, log error, move to next
  - Never delete messages — only set `is_archived_msg = TRUE`
- [ ] Logs: `{event: "compaction", conversation_id: "...", messages_archived: 30, summary_tokens: 450}`
- [ ] Verification: seed a conversation with 60+ messages → run summarizer → confirm summary message exists, old messages have `is_archived_msg = TRUE`, frontend LLM context query returns only summary + recent messages, frontend scroll still shows all messages

### 6.7 — Deploy & Validate All Crons
- [ ] Deploy all 6 timer-triggered functions to `agent-ochuko-functions` Function App
- [ ] Verify each function appears in Azure Portal → Function App → Functions list
- [ ] Verify timer triggers are registered in Azure Portal → Function App → App files → `host.json`
- [ ] Manually trigger each function from Azure Portal "Test/Run" to confirm end-to-end
- [ ] Check Application Insights for all log entries from each cron

**✅ Milestone**: All background automation running. Token budgets reset daily, agent quotas reset monthly, usage stats materialized hourly, stale conversations archived, model deployments monitored for expiry, and long conversations compacted by GPT-o4-mini — all without any manual intervention or code changes.

## Phase 7 — Production Hardening
**Duration**: 3–5 days | **Depends on**: All previous phases complete and functional

### 7.1 — Observability & Structured Logging (Day 1)
- [ ] Integrate `opencensus-ext-azure` or `azure-monitor-opentelemetry` into FastAPI
- [ ] Every request logs a structured trace: `{request_id, user_id, method, path, status_code, latency_ms, model_used, tokens_in, tokens_out}`
- [ ] Every exception logs full stack trace + request context to Application Insights Exceptions table
- [ ] Custom metrics to Application Insights:
  - `chat.stream.ttft_ms` — time to first token (P50, P95, P99)
  - `chat.stream.total_ms` — total stream duration
  - `agent.job.duration_ms` — by agent type
  - `agent.job.failure_rate` — by agent type
  - `budget.utilization_pct` — per user, per day
  - `model_router.decision` — count by mode (think/solve/discuss/nano)
- [ ] Azure Function workers: same structured logging pattern, each job logs `{job_id, type, status, duration_ms, error}`
- [ ] Set up Application Insights Alerts:
  - **Error rate > 5% over 5 minutes** → email admin
  - **P95 TTFT > 3 seconds** → email admin
  - **Any cron function fails** → email admin
  - **Budget utilization > 80% globally** → email admin

### 7.2 — Health Check & Readiness (Day 1)
- [ ] `GET /health` — returns `200 OK` with:
  ```json
  {
    "status": "healthy",
    "version": "1.0.0",
    "commit": "abc1234",
    "checks": {
      "supabase": "ok",
      "azure_openai": "ok",
      "azure_queue": "ok",
      "azure_app_config": "ok",
      "azure_blob": "ok"
    }
  }
  ```
- [ ] Each check: lightweight ping/HEAD — timeout 2 seconds per service, mark "degraded" if any fail
- [ ] If Supabase OR Azure OpenAI is down → return `503` with degraded status
- [ ] Azure Container Apps uses `/health` as the **liveness probe** (restart if unhealthy for 3 consecutive checks)
- [ ] Add a separate `GET /ready` — returns 200 only if `load_config()` has completed and Supabase client is initialized (used as **readiness probe** — Container Apps won't route traffic until ready)

### 7.3 — CI/CD Pipelines (Day 1–2)
- [ ] **Backend pipeline** (`.github/workflows/backend-deploy.yml`):
  1. Trigger: push to `main` with changes in `backend/**`
  2. Run `pytest` (unit tests) — fail pipeline if any test fails
  3. Run `ruff check` (linting) + `ruff format --check` (formatting)
  4. Build Docker image, tag with `git SHA`
  5. Push to private Docker Hub registry
  6. Deploy to Azure Container Apps
  7. Wait for health check to pass on new revision
- [ ] **Frontend pipeline** (`.github/workflows/frontend-deploy.yml`):
  1. Trigger: push to `main` with changes in `frontend/**`
  2. `npm ci` → `npm run build` → `npm run lint`
  3. Deploy `dist/` to Azure Storage Blob static website container (`$web` on `agentochukostore`)
- [ ] **Admin pipeline** (`.github/workflows/admin-deploy.yml`):
  1. Same as frontend but deploys to Azure Storage Blob static website container (`$web` on `agentochukoadmin`)
- [ ] **Functions pipeline** (`.github/workflows/functions-deploy.yml`):
  1. Trigger: push to `main` with changes in `functions/**`
  2. Run function-specific unit tests
  3. Deploy to Azure Function App using `func azure functionapp publish`
- [ ] All pipelines use GitHub Secrets for: `AZURE_CREDENTIALS`, `DOCKER_USERNAME`, `DOCKER_PASSWORD`

### 7.4 — Container Deployment Strategy (Day 2)
- [ ] **Blue/green (revision-based)** on Azure Container Apps:
  - New revision deployed with 0% traffic initially
  - Health check passes → shift 100% traffic to new revision
  - Old revision kept for 1 hour as instant rollback target
  - If health check fails → new revision deactivated, old revision keeps serving
- [ ] Set Container App scaling:
  - Min replicas: 1 (always warm — no cold start for users)
  - Max replicas: 3 (handle burst traffic from concurrent users)
  - Scale rule: HTTP concurrent requests > 20 per replica → scale out
- [ ] Resource limits: 0.5 vCPU, 1 GB RAM per replica (upgrade from Phase 0's 0.25 vCPU if needed based on load test results)

### 7.5 — Security Hardening Review (Day 2–3)
- [ ] **RLS policy audit**: for every table, test with 5 role-specific JWTs (guest, user, power_user, admin, superadmin):
  - `guest` can ONLY read shared conversations and their messages — nothing else
  - `user` can ONLY read/write their own conversations, messages, jobs
  - `user` CANNOT read another user's data even with a hand-crafted query
  - `admin` can read all data, manage users, but CANNOT modify `audit_log`
  - `superadmin` has full admin access + can modify App Configuration via admin endpoints
- [ ] **JWT validation hardening**:
  - Verify `iss` (issuer) matches Supabase project URL
  - Verify `aud` (audience) is correct
  - Verify `exp` is not in the past
  - Reject tokens signed with `none` algorithm (JWT alg confusion attack)
  - Cache JWKS keys for 1 hour, refresh on miss
- [ ] **CORS strict audit**:
  - `ALLOWED_ORIGINS` contains ONLY: `https://agentochukostore.z1.web.core.windows.net` (chat), `https://agentochukoadmin.z1.web.core.windows.net` (admin), `http://localhost:5173` (dev)
  - No wildcard `*` — ever
  - `allow_credentials = True` — required for JWT cookie flow
- [ ] **Rate limiting** (if not already in Phase 3):
  - Per-IP: 100 requests/minute (prevent scraping/abuse)
  - Per-user: 30 streaming requests/minute (prevent token burning)
  - Implementation: `slowapi` middleware using in-memory storage (`memory://`), relying on Azure Container Apps sticky sessions (session affinity) for replica consistency.
- [ ] **File upload security**:
  - Validate MIME type server-side before generating presigned URL
  - Max file size enforced: `admin_settings.max_file_size_mb` (default 10MB)
  - Blob container access: `uploads` = private (presigned URL only), `generated` = private, `exports` = private
  - No executable file types allowed (`.exe`, `.sh`, `.bat`, `.cmd`, `.ps1`, `.js`)
- [ ] **Headers hardening** on FastAPI:
  - `X-Content-Type-Options: nosniff`
  - `X-Frame-Options: DENY`
  - `Strict-Transport-Security: max-age=31536000; includeSubDomains`
  - `Content-Security-Policy: default-src 'self'` (on frontend via SWA config)
- [ ] **Dependency audit**: run `pip audit` on `requirements.txt` — resolve any CVEs before production

### 7.6 — Load Testing (Day 3–4)
- [ ] Tool: `locust` or `k6` — script simulates realistic user behavior
- [ ] **Test scenario 1 — Concurrent streaming** (primary):
  - 50 virtual users, each sends 1 chat message every 30 seconds
  - Measure: TTFT (P50, P95, P99), total stream time, error rate
  - Pass criteria: P95 TTFT < 2 seconds, error rate < 1%
- [ ] **Test scenario 2 — Agent burst**:
  - 20 virtual users, each submits 1 OCR job simultaneously
  - Measure: time from 202 response to job `status = 'done'`
  - Pass criteria: 95% of jobs complete within 30 seconds
- [ ] **Test scenario 3 — Budget exhaustion under load**:
  - 10 users hit their daily token budget simultaneously mid-stream
  - Verify: stream terminates gracefully with 429, no partial writes, no double-deduction
- [ ] **Test scenario 4 — Mixed load**:
  - 30 chat users + 10 agent users + 5 admin dashboard users concurrently
  - Measure: no cross-contamination (user A never sees user B's data), no RLS bypass under load
- [ ] Document results in `docs/05_load_test_results.md`: latency distributions, throughput, failure modes, bottleneck analysis
- [ ] If bottlenecks found: tune Container App replica count, adjust vCPU/RAM, optimize slow Supabase queries (check `pg_stat_statements`)

### 7.7 — Final Cost Validation (Day 4)
- [ ] Run the full system for 48 hours with simulated usage (10–20 active test accounts)
- [ ] Check Azure Cost Management:
  - Azure OpenAI: total token spend — project monthly at 100 users
  - Container Apps: vCPU-seconds and memory-seconds consumed
  - Functions: total executions
  - Storage: blob size + queue transactions
  - All other services: confirm free tier not exceeded
- [ ] Compare against the cost analysis table in Section 14 of the implementation plan
- [ ] If monthly projection exceeds $12: identify the cost driver, adjust token budgets or agent quotas accordingly
- [ ] Document in `docs/06_cost_validation.md`

### 7.8 — Error Handling & Graceful Degradation (Day 4–5)
- [ ] **Azure OpenAI down**: FastAPI returns `503 Service Temporarily Unavailable` with `Retry-After: 30` header — frontend shows "AI service is temporarily unavailable, please try again shortly"
- [ ] **Supabase down**: health check marks degraded — existing streams can complete (in-flight), new requests get 503
- [ ] **Azure Queue down**: agent endpoints return 503 instead of silently failing — frontend shows clear error
- [ ] **Hugging Face all keys exhausted**: `image_gen_worker` writes `status = 'failed'` with error "Image generation temporarily unavailable" — frontend shows human-readable message, not stack trace
- [ ] **Groq Whisper down**: `speech_stt_worker` falls back to Azure Speech STT (if configured) or fails gracefully
- [ ] **Frontend SSE stream breaks mid-response**: `useStream.ts` implements auto-reconnect — if stream drops, show partial response with "(Response interrupted — click to retry)" button
- [ ] **Supabase Realtime drops**: `useJob.ts` implements fallback polling — if no Realtime update received within 15 seconds for a pending job, poll `GET /v1/jobs/{id}` every 10 seconds
- [ ] All error responses follow a consistent JSON shape:
  ```json
  {
    "error": {
      "code": "BUDGET_EXHAUSTED",
      "message": "Daily token budget exhausted. Resets at midnight UTC.",
      "retry_after": 3600
    }
  }
  ```

**✅ Milestone**: System is production-grade. Observable, tested under load, secured at every layer, cost-validated, CI/CD automated, graceful under failure. Ready for real users.
