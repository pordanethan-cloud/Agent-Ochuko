# Setup Guide: Cloudflare R2 Bucket Setup (Wrangler CLI & Web Interface)

This guide walks you through creating a new Cloudflare R2 bucket named `agent-ochuko-storage`, generating S3-compatible API credentials, and configuring public access to replace Azure Blob Storage.

---

## 1. Setting up the R2 Bucket

You can set up your bucket using either the Command Line (Wrangler CLI) or the Cloudflare Dashboard Web Interface.

### Option A: Using Wrangler CLI (Command Line)

If you have `wrangler` installed globally or want to use `npx`, follow these commands:

1. **Log in to Cloudflare**:
   ```bash
   npx wrangler login
   ```
   *This will open a browser window asking you to authorize Wrangler with your Cloudflare account.*

2. **Create the R2 Bucket**:
   Create the bucket with the recommended name:
   ```bash
   npx wrangler r2 bucket create agent-ochuko-storage
   ```

3. **Verify the Bucket**:
   Confirm your bucket was created successfully:
   ```bash
   npx wrangler r2 bucket list
   ```

---

### Option B: Using the Cloudflare Web Interface

1. Log in to your [Cloudflare Dashboard](https://dash.cloudflare.com/).
2. In the left-hand sidebar, navigate to **R2** (or click on **Object Storage**).
3. Click the **Create bucket** button in the top right.
4. **Configure your bucket**:
   - **Bucket name**: Enter `agent-ochuko-storage`.
   - **Location hint**: Keep it as **Automatic** (or select the region closest to your Azure Functions deployment).
5. Click **Create bucket** at the bottom.

---

## 2. Generating API Credentials (Access Key & Secret Key)

To connect Ochuko backend/workers to your new bucket, you must generate S3-compatible credentials.

1. Go to the Cloudflare Dashboard and click on **R2** in the sidebar.
2. In the right-hand sidebar or menu of the R2 main screen, click on **Manage R2 API Tokens**.
3. Click the **Create API token** button.
4. **Configure the Token**:
   - **Token name**: Choose a descriptive name (e.g., `Ochuko-R2-Token`).
   - **Permissions**: Select **Admin Read & Write** (or **Edit** permissions).
   - **Specify bucket(s)**: 
     - Choose **Apply to specific buckets only** and select `agent-ochuko-storage`.
     - *(Alternatively, choose **Apply to all buckets** if you want the token to access multiple buckets).*
   - **TTL (Expiry)**: Set to **Forever** (or a duration of your choice).
5. Click **Create API Token**.
6. **Copy and Save the credentials immediately**:
   > [!WARNING]
   > For security reasons, Cloudflare will only show the Secret Access Key **once**. Copy these values before leaving the page:
   - **Access Key ID** (Corresponds to `R2_ACCESS_KEY_ID`)
   - **Secret Access Key** (Corresponds to `R2_SECRET_ACCESS_KEY`)
   - **Jurisdiction-specific Endpoint** (Corresponds to `R2_ENDPOINT`):
     - Copy the **S3 API** URL (e.g., `https://<account-id>.r2.cloudflarestorage.com`).
     - Remove any trailing bucket name from the URL if present; it should be just the host URL.

## 3. Configuring Public Access

Because images and audio files generated by Ochuko need to be accessed directly in the chat UI, the bucket's contents must be publicly readable.

1. Go to your **R2** Dashboard and select the `agent-ochuko-storage` bucket.
2. Click on the **Settings** tab.
3. Scroll down to the **Public Access** section.
4. Choose **one** of the following options:

### Option 1: Enable the R2.dev Subdomain (Quickest)
- Under **Public Access**, look for the **r2.dev Subdomain** section.
- Click **Allow Access** (or **Enable**).
- Copy the public URL domain (e.g., `https://pub-xxxxxxxxxxxxxx.r2.dev`).
- This URL base will be your `R2_PUBLIC_DOMAIN`.

### Option 2: Connect a Custom Domain (Recommended for Production)
- Under **Public Access**, click **Connect Domain**.
- Enter your custom subdomain (e.g., `files.yourdomain.com`).
- Click **Continue** and follow the prompts to configure the DNS record (if your domain is managed by Cloudflare, it will configure it automatically).
- This custom domain (e.g., `https://files.yourdomain.com`) will be your `R2_PUBLIC_DOMAIN`.

Once you have your credentials, update your configuration files with the new values.

### 1. Update [backend/.env](file:///c:/Users/T14 GEN 5/Documents/WORK AND PLAN/AZURE SYSTEM-AUTH AT SCALE/agent-ochuko/backend/.env)

```env

R2_SECRET_ACCESS_KEY=your_new_secret_access_key

R2_PUBLIC_DOMAIN=https://your_public_subdomain.r2.dev
```

```json
    "R2_ACCESS_KEY_ID": "your_new_access_key_id",
    "R2_SECRET_ACCESS_KEY": "your_new_secret_access_key",
    "R2_ENDPOINT": "https://your_cloudflare_account_id.r2.cloudflarestorage.com",
    "R2_BUCKET_NAME": "agent-ochuko-storage",
    "R2_PUBLIC_DOMAIN": "https://your_public_subdomain.r2.dev"

## 5. Production Setup: Azure App Configuration & Key Vault

In a production environment, you should avoid storing sensitive secrets directly in Azure App Configuration as plaintext. Instead, store them in **Azure Key Vault** and reference them in **Azure App Configuration**.

### 1. Store Secrets in Azure Key Vault

Run these Azure CLI commands to store the sensitive R2 credentials in your Key Vault:

```bash
# 1. Store the Access Key ID
az keyvault secret set --vault-name <your-keyvault-name> --name "R2-Access-Key-ID" --value "33fbceeefd93cef7bda84347d5361e88"

# 2. Store the Secret Access Key
az keyvault secret set --vault-name <your-keyvault-name> --name "R2-Secret-Access-Key" --value "5291d7c89d8f862c787eb8e62cd4002648de3dd67e5b2a688745176929307457"
```

### 2. Set Configuration Settings in Azure App Configuration

Run these Azure CLI commands to set the non-sensitive configuration values directly in App Configuration:

```bash
# 1. Set the Endpoint URL
az appconfig kv set --name agent-ochuko-appconfig --key R2_ENDPOINT --value "https://3aabaa1c09b9c0da240f1ae5ed8d8478.r2.cloudflarestorage.com" --label production --yes

# 2. Set the Bucket Name
az appconfig kv set --name agent-ochuko-appconfig --key R2_BUCKET_NAME --value "agent-ochuko-storage" --label production --yes

# 3. Set the Public Domain URL
az appconfig kv set --name agent-ochuko-appconfig --key R2_PUBLIC_DOMAIN --value "https://pub-e278636d72f14ae3940a3be367b1ff15.r2.dev" --label production --yes
```

### 3. Create Key Vault References in Azure App Configuration

To link the secrets from your Key Vault into App Configuration, create **Key Vault References**:

1. Open the **Azure Portal** and navigate to your **Azure App Configuration** instance (`agent-ochuko-appconfig`).
2. Go to **Configuration explorer** under the **Operations** section.
3. Click **Create** ➔ **Key Vault reference**.
4. Configure the first reference for **`R2_ACCESS_KEY_ID`**:
   - **Key**: `R2_ACCESS_KEY_ID`
   - **Label**: `production`
   - **Key Vault**: Select your Key Vault name
   - **Secret**: Select `R2-Access-Key-ID`
5. Click **Apply**.
6. Repeat the steps to configure the reference for **`R2_SECRET_ACCESS_KEY`**:
   - **Key**: `R2_SECRET_ACCESS_KEY`
   - **Label**: `production`
   - **Key Vault**: Select your Key Vault name
   - **Secret**: Select `R2-Secret-Access-Key`
7. Click **Apply**.

### 4. Enable Managed Identities

Make sure both your **FastAPI Web App** and **Azure Function App** have System-Assigned Managed Identity enabled and are granted the **Key Vault Secrets User** role on the Key Vault. This allows the app runtime to decrypt the references and retrieve the actual credential values automatically.
