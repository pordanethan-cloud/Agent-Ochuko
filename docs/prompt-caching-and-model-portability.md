# Prompt caching, capabilities, and multi-model portability
 
## Current treatment of tools and capabilities
 
The reliability change separates persona instructions from tool capability definitions:
 
- **Discuss/Nano:** sends the compact persona, `search_web` for non-chitchat, tools matched by the current intent, and tools already activated in the conversation.
- **Think/Solve:** sends the compact core persona and the complete compact tool-schema set for non-chitchat tasks.
- **Pure greetings:** send no tool schemas unless the conversation already activated a tool.
- **Capability registry narrative:** is no longer appended to every Think/Solve system prompt. It duplicated information already present in the selected tool schemas and added approximately 790 input tokens per turn.
 
The model still receives the actionable information it needs: tool name, purpose, arguments, and selection policy. The removed text was a second prose description of the same capabilities.
 
## Prompt caching is not model memory
 
Prompt caching reuses computation for an identical prompt prefix. It does not let the application send capabilities once and omit them forever.
 
Every request must remain complete and correct if the cache is empty. The provider may reuse the fixed prefix internally when the same persona, capability profile, tool definitions, and ordering are sent again.
 
For a conversation that changes from chat to a task:
 
1. Chat turns use the small `chat` profile.
2. The first task turn selects a capability profile and is normally a cache write/miss.
3. Later turns with the same profile can reuse that cached prefix.
4. Switching to another profile can use a different existing cache entry or create one.
 
The cache is an optimization only. Supabase conversation history, summaries, memory, and tool results remain the source of truth.
 
## Azure/OpenAI behavior
 
Azure prompt caching is automatic on supported models. The current `gpt-5.4` deployment also supports extended retention:
 
```python
stream_kwargs["prompt_cache_key"] = (
    f"ochuko:{deployment}:{PERSONA_VERSION}:{capability_profile.version}"
)
stream_kwargs["prompt_cache_retention"] = "24h"
```
 
Important constraints:
 
- The request must contain at least 1,024 prompt tokens.
- The first 1,024 tokens must be identical.
- Further hits are calculated in 128-token increments.
- A difference in the prefix causes a miss.
- Tool use includes both the message input and tool definitions in the cached material.
- Cache hits are reported as `cached_tokens` in prompt-token usage details.
- In-memory retention normally expires after inactivity; supported Azure models can use `24h` retention.
 
Do not pad a short prompt merely to reach 1,024 tokens. A 400-token prompt is already inexpensive. Caching becomes valuable naturally for capability-heavy task profiles, long reusable instructions, or large tool sets.
 
Official reference: https://learn.microsoft.com/en-us/azure/ai-foundry/openai/how-to/prompt-caching
 
## Claude behavior
 
Claude uses the same prefix-reuse principle but exposes provider-specific controls:
 
- Top-level automatic `cache_control` can move the cache breakpoint as a conversation grows.
- Explicit cache breakpoints can be attached to cacheable content blocks.
- Common TTLs are five minutes and one hour.
- Usage reports separate `cache_creation_input_tokens` and `cache_read_input_tokens`.
- Claude evaluates the prompt prefix in this order: `tools`, `system`, then `messages`.
 
Example adapter output:
 
```python
request = {
    "model": model,
    "max_tokens": max_tokens,
    "cache_control": {"type": "ephemeral", "ttl": "1h"},
    "tools": capability_profile.anthropic_tools,
    "system": [
        {"type": "text", "text": stable_persona},
        {"type": "text", "text": capability_profile.policy},
        {"type": "text", "text": dynamic_context},
    ],
    "messages": messages,
}
```
 
The exact SDK shape should be generated by the Claude adapter and covered by contract tests, because supported cache-control fields can vary by model and API version.
 
Official reference: https://docs.anthropic.com/en/docs/build-with-claude/prompt-caching
 
## Recommended prompt layout
 
Keep reusable content first and volatile content last:
 
```text
1. Stable identity, safety, context, and response rules
2. Versioned capability-profile policy
3. Canonically ordered tool definitions
4. Dynamic user name, timezone, and request-specific environment
5. Conversation summary and recent turns
6. Latest user message
```
 
The current implementation concatenates user/time context into the system string. For stronger cache reuse, compile them into a later system block:
 
```python
input_messages = [
    {"role": "system", "content": stable_persona},
    {"role": "system", "content": capability_profile.policy},
    {"role": "system", "content": dynamic_context},
    *conversation_messages,
]
```
 
Never put timestamps, user names, request IDs, conversation IDs, or mutable memory before the stable prefix.
 
## Replace a growing tool registry with finite profiles
 
The present Discuss/Nano selector remembers activated tools and continues sending them. That preserves capability continuity, but the schema set can grow and create many unique cache prefixes.
 
A more cacheable design uses a small set of versioned profiles:
 
| Profile | Tools | Typical use |
|---|---|---|
| `chat-v1` | none | greetings and ordinary conversation |
| `web-v1` | `search_web` | recent or externally verifiable information |
| `creator-v1` | `generate_file`, `generate_image`, `read_file` | documents, images, uploads |
| `code-v1` | `run_code_agent`, `read_file` | computation, coding, data work |
| `agent-v1` | all supported tools | complex Think/Solve execution |
 
The router selects one profile per turn. Tool lists must use canonical ordering and stable JSON serialization. A profile change is intentional cache segmentation, not a bug.
 
Do not create a new profile for every combination of tools. A small finite set produces reusable cache entries across conversations.
 
## Provider-neutral architecture
 
Use an internal plan that contains no Azure- or Claude-specific request fields:
 
```python
@dataclass(frozen=True)
class PromptPlan:
    persona_version: str
    stable_system: tuple[str, ...]
    dynamic_system: tuple[str, ...]
    capability_profile: CapabilityProfile
    messages: tuple[ChatMessage, ...]
    max_output_tokens: int
 
 
class ModelAdapter(Protocol):
    async def stream(self, plan: PromptPlan) -> AsyncIterator[ModelEvent]:
        ...
```
 
Provider adapters translate the plan:
 
```text
AzureResponsesAdapter
  -> Responses input roles
  -> OpenAI function-tool schema
  -> prompt_cache_key / prompt_cache_retention
  -> cached_tokens metrics
 
ClaudeMessagesAdapter
  -> Claude system content blocks
  -> Anthropic tool schema
  -> cache_control breakpoints
  -> cache_creation_input_tokens / cache_read_input_tokens
```
 
Application code should consume normalized events:
 
```python
TextDelta(text)
ToolCallStarted(id, name, arguments)
ToolCallCompleted(id, result)
Usage(input_tokens, output_tokens, cached_input_tokens)
Completed(provider_response_id)
Failed(code, safe_message)
```
 
This prevents `chat.py` and the frontend SSE contract from depending directly on one provider's stream event format.
 
## Conversation state across models
 
Do not make `previous_response_id`, a Claude message ID, or any provider thread the canonical conversation:
 
- Provider response chaining is not portable.
- A conversation can move from GPT to Claude only if the application owns the normalized history.
- Provider IDs can be stored as optional acceleration metadata.
- On a model switch, rebuild input from the database summary plus recent messages.
- Persistent memory must be provider-neutral structured data, not hidden provider state.
 
Recommended stored fields:
 
```json
{
  "conversation_id": "uuid",
  "summary": "provider-neutral summary",
  "recent_messages": [],
  "memory": {},
  "last_provider": "azure-openai",
  "provider_state": {
    "azure_response_id": "optional",
    "claude_message_id": "optional"
  }
}
```
 
## Cache keys and versioning
 
Cache namespaces should identify only stable material:
 
```python
cache_key = ":".join([
    "ochuko",
    provider,
    model_family,
    persona_version,
    capability_profile.version,
])
```
 
Do not include `user_id` or `conversation_id` when the prefix contains only shared static instructions. Include a tenant/security boundary if separate tenants must never share cache routing.
 
Bump a version when any of these change:
 
- Persona semantics
- Tool name, description, or schema
- Capability policy
- Safety policy
- Stable block ordering
 
## Metrics
 
Record these per provider, model, and capability profile:
 
- Prompt tokens
- Cached input tokens
- Cache creation tokens
- Cache hit ratio
- Time to first token
- Total response latency
- Input cost and cached-input cost
- Tool-call success rate
- Context-correctness evaluation score
 
Prompt caching should be enabled only after usage metrics are captured. Otherwise reduced billing or latency cannot be verified.
 
## Recommended rollout
 
1. Keep the current compact prompts and selected tool schemas.
2. Introduce `PromptPlan`, `CapabilityProfile`, and `ModelAdapter` without changing behavior.
3. Split stable and dynamic system blocks.
4. Add Azure `prompt_cache_key="ochuko:azure:<model>:<persona>:<profile>"` and `prompt_cache_retention="24h"`.
5. Log `cached_tokens` and compare cache hit rate by profile.
6. Replace the growing registered-tool set with finite profiles.
7. Add the Claude adapter and provider contract tests.
8. Run the same context and tool-use evaluation suite against every model.
 
This preserves reliability while allowing model choice to expand without rewriting conversation storage, tool execution, or frontend streaming.
